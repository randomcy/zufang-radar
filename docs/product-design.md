# 产品设计文档

## 一、定位

**一句话**：一个"反中介"的租房决策助手，让年轻人靠真实数据和真实声音做选择。

**核心价值主张**：
- 传统房源平台告诉你房子的物理属性
- 租房雷达告诉你这套房子是否真的适合你 + 住进去会是什么体验

## 二、目标用户

- 一线城市 22-30 岁年轻人
- 第一次或第二次独立租房
- 在小红书上搜过"XX 小区怎么样""租房避坑"的人

## 三、用户旅程

```
首页 / Landing
   │
   ├──► [我还不知道想要什么]      ──► 场景 A：人格测试 ──┐
   │                                                       │
   └──► [我在几个候选里纠结]       ──► 场景 B：决策助手 ──┤
                                                            │
                                                            ▼
                                              偏好画像 / 候选排序
                                                            │
                                                            ▼
                                              功能二：通勤地图筛选
                                                            │
                                                            ▼
                                              功能三：小区体检报告
                                                            │
                                                            ▼
                                                        做出决策
```

故事主线：**自我认知 → 候选筛选 → 决策验证**

## 四、功能详解

### 功能一·场景 A：租房人格测试

**目标**：用户对自己偏好模糊时，通过 conjoint 反推隐性偏好。

**核心 attribute（6 个，每个 2-3 个 level）**：

| Attribute | Levels |
|-----------|--------|
| 月租价格 | 3000 / 5000 / 7500 / 10000 |
| 房型 | 合租单间 / 整租开间 / 一居室 / 两居室 |
| 通勤时间 | 20 分钟内 / 40 分钟内 / 60 分钟内 |
| 装修档次 | 老破普装 / 普通装修 / 精装 / 网红装修 |
| 小区品质 | 老破小无物业 / 普通小区 / 优质物业 |
| 房龄/楼栋 | 老破小 / 板楼 / 新塔楼 |

**附加 yes/no 偏好（不进 conjoint 主流程，作为硬筛选）**：
- 必须独卫
- 必须能养宠物
- 必须有阳台
- 必须有电梯
- 必须近地铁（步行 10 分钟内）

**题目数量**：8 道（每题展示 2 套虚构房源，让用户二选一）

**算法**：
1. 每个 attribute level 初始 utility = 0
2. 用户每次选择 A，则 A 中所有 level 的 utility +1，B 中所有 level 的 utility -1
3. 8 道题后，按 attribute 归一化得到每个 level 的 part-worth utility
4. attribute 权重 = (max level utility - min level utility) within attribute
5. 全部权重归一化为百分比，输出雷达图

**输出**：
- 偏好雷达图（6 个 attribute 的权重）
- 排序：最在乎的 3 个 + 可妥协的 3 个
- 租房人格标签：如「精致独居型」「通勤优先型」「极致性价比型」「品质生活型」

### 功能一·场景 B：决策助手

**目标**：用户已有 3-5 个真实候选，纠结选哪个。

**输入**：每个候选填核心字段（名称 + 月租 + 通勤时间 + 装修 + 房型 + 小区品质）

**流程**：
1. 系统自动生成所有两两组合（C(5,2)=10 题，C(4,2)=6 题，C(3,2)=3 题）
2. 卡片左右对比，用户点击选择
3. **Bradley-Terry 模型** 计算每个候选的实力分数
4. 输出排序 + 解释

**关键 wow 时刻**：
> "在 10 次比较中，你 8 次选了通勤更短的房源，说明通勤对你权重最高。基于这个洞察，我们推荐你选 XX。"

### 功能二：通勤地图筛选

**输入**：
- 公司位置（地图选点 / 搜索地址）
- 最大可接受通勤时间（30 / 45 / 60 分钟）
- 通勤方式（地铁 / 地铁+步行）

**输出**：
- 地图上标记所有符合条件的地铁站
- 点击站点 → 展示该站周边 mock 房源列表
- 房源卡片显示：图片、价格、通勤时间、匹配度（基于场景 A 的偏好画像）

**技术**：高德地图 JS API + 路径规划 API

### 功能三：小区体检报告

**触发**：从房源卡片点击"看小区评价" 或 从地图点击小区图标

**报告内容**：
1. **总体评分**（5 星）+ "适合人群" 标签
2. **优点 TOP 3**（卡片式）
3. **缺点 TOP 3**（卡片式）
4. **分项雷达图**：噪音、隔音、物业、安全、周边配套、性价比
5. **真实声音**：3 条引用帖子（标题 + 摘要 + 虚构作者）
6. **与你的匹配度**（基于场景 A 偏好画像）

**Mock 数据策略**：
- 用 GPT-4 生成每个小区 20-30 条"小红书风格"帖子
- 用 LLM 对帖子做提取：优缺点、评分、关键词
- 全流程在 build 阶段预生成，运行时直接读 JSON

## 五、面试讲稿核心论点

1. **为什么是这三个功能**：自我认知 → 筛选 → 验证，完整闭环
2. **为什么是小红书做这个**：UGC 内容是壁垒，贝壳没有
3. **为什么用 conjoint**：解决"用户说不清自己要什么"的根本问题
4. **商业逻辑**：租房是高客单价决策，是小红书种草-决策闭环里最薄弱的一环

## 六、不做什么（明确边界）

- ❌ 不做真实房源抓取（数据合规风险 + 时间不够）
- ❌ 不做用户系统/登录（demo 不需要）
- ❌ 不做移动端适配（PC 端足够展示）
- ❌ 不做后端服务（纯前端 + JSON 数据）
